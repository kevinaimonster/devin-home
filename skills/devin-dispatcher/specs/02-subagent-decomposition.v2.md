## 问题陈述

当前 devin-dispatcher 在单一上下文中执行所有工作：探索代码 → 制定方案 → 修改代码 → 验证。在复杂任务中（触及 5+ 文件、跨多个模块），探索阶段读取的大量文件内容**污染**了执行阶段的上下文窗口，导致：

1. **上下文耗尽** — 探索阶段读了 20 个文件，执行阶段只剩 30% 的上下文空间
2. **注意力分散** — 上下文中充斥的无关文件内容干扰了修改代码时的推理质量

Claude Code 的核心架构是：**主 agent 协调决策，sub-agent 执行具体工作**。主 agent 从不自己读大量文件。分解的核心价值在于：

- **上下文隔离** — 探索产出的大量文件内容留在探索 Agent 上下文中，不进入 Devin 主体
- **关注点分离** — 每个实现 Agent 只关注自己负责的子任务，上下文更专注，推理质量更高
- **故障隔离** — 某个子任务失败不影响其他已完成的子任务

> **关于并发执行**：Claude Code 的 Agent 工具标记为 `isConcurrencySafe: true`，多个 Agent 调用在同一条消息中发出时，底层确实会并发执行。但这种并发带来的时间收益有限（受限于 API 速率和模型推理时间），不应作为分解的主要动机。分解的决策应完全基于上下文管理需求，而非加速需求。

## 需求描述

### 不分解阈值

在进入分解判断之前，先检查是否值得分解：

**不分解条件（满足任一则跳过分解，Devin 直接在主上下文中执行）**：
- 预期总修改量 < 50 行（含新增 + 删除）
- 预期子任务数 < 3
- 所有修改集中在同一个文件
- 修改之间有紧密的数据/接口依赖（如改 API 返回类型 + 改调用方）
- 用户明确说"简单改一下"或修改量很小

> 判断依据：分解本身有成本（准备 prompt、等待 agent 返回、集成检查）。对于小任务，这些成本超过了上下文隔离的收益。

### 探索阶段使用探索 Agent（Phase 1.3 增强）

当项目知识（DEVIN.md / CLAUDE.md）不足以定位要修改的文件时：

**现有行为**（保留作为回退）：Devin 自己执行 Glob / Grep / Read

**新增行为**（优先）：派一个探索 Agent

#### 何时使用探索 Agent vs 自己探索

| 场景 | 方式 |
|------|------|
| 需要定位未知代码位置（"找到处理 X 的代码"） | 探索 Agent |
| 需要读取 3+ 个文件来理解架构 | 探索 Agent |
| 已知文件路径，只需确认内容 | 自己读 |
| DEVIN.md 已有足够的领域 → 文件映射 | 自己读 |

#### 探索 Agent 调用方式

```
使用 Agent 工具，prompt 包含：
- 项目路径
- 需求关键词（如 "支付回调处理"）
- 需要定位的内容（如 "找到处理支付回调的所有文件，以及相关的类型定义和路由"）
- 明确要求按下方"探索 Agent 输出格式"返回结构化结果

探索 agent 约束：
- 只使用只读工具（Glob / Grep / Read）
- 必须按结构化模板返回结果
```

#### 探索 Agent 输出格式模板

探索 Agent 的 prompt 中必须包含以下输出格式要求：

```markdown
请按以下格式返回结果：

## 模块结构
{描述涉及的模块层次关系，如 "路由层 → Controller → Service → Repository"}

## 关键文件（按修改优先级排序）

### 1. {文件路径}
- **角色**: {该文件在架构中的角色}
- **关键接口/类型**:
  ```typescript
  // 只摘录接口签名、类型定义、函数签名，不要完整文件
  interface PaymentCallback {
    orderId: string;
    status: 'success' | 'failed';
  }
  ```
- **修改切入点**: {具体描述从哪里开始改、改什么}

### 2. {文件路径}
...

## 共享依赖
- **共享类型定义文件**: {列出被多个模块引用的类型定义文件}
- **共享状态/配置**: {列出被多个模块共享的状态管理、配置文件}
- **跨文件 import 关系**: {列出文件间的 import 依赖}

## 风险提示
{列出可能影响分解判断的因素，如 "文件 A 和文件 B 共享 UserContext 类型，修改时需注意"}
```

#### 数据流：探索 Agent → Devin → 实现 Agent

这是分解策略的核心数据流，解决"上下文准备悖论"：

```
┌─────────────┐     结构化报告          ┌──────────┐     精简 prompt      ┌─────────────┐
│  探索 Agent  │ ──────────────────────→ │  Devin   │ ──────────────────→ │  实现 Agent  │
│  (只读)      │  - 模块结构             │  (主体)   │  - 子任务描述       │  (可写)      │
│             │  - 关键接口签名          │          │  - 目标文件路径      │             │
│             │  - 共享依赖分析          │          │  - 接口签名片段      │             │
│             │  - 修改切入点            │          │  - 项目规范         │             │
└─────────────┘                         └──────────┘                     └─────────────┘
     ↑                                       │
     │                                       │ 不读完整文件
     │ 读了 20 个文件                         │ 只提取 Agent 报告中
     │ 内容留在 Agent 上下文                   │ 的关键片段
     │ 不进入 Devin 上下文                     ↓
```

**关键原则**：
1. 探索 Agent 负责大量读文件，输出结构化报告（接口签名、类型定义、修改切入点）
2. Devin 主体**只读探索 Agent 的返回结果**（即 Agent 工具的 tool_result），不自己再去读那些文件
3. Devin 从探索结果中**提取关键片段**（接口签名、类型定义），拼入实现 Agent 的 prompt
4. 实现 Agent 拿到的是精简的、与子任务直接相关的上下文，不是完整的文件内容

**反模式**（禁止）：
- Devin 收到探索结果后，自己再用 Read 工具去读探索 Agent 提到的文件 → 上下文照样被污染
- Devin 让实现 Agent 自己去搜索和读文件 → 实现 Agent 效率低且可能找错位置

### 执行阶段按需分解（Phase 3.2 增强）

#### 独立性判断标准（Checklist）

修改清单中有 **3 个以上互相独立的子任务**（满足不分解阈值后）时启用分解。

**独立性检查 Checklist** — 两个子任务必须同时满足以下所有条件才视为"独立"：

- [ ] **无文件重叠** — 不修改同一个文件
- [ ] **无 import 关系** — 子任务 A 修改的文件不 import 子任务 B 修改的文件，反之亦然
- [ ] **无共享类型修改** — 不同时修改同一个类型定义（如 `types.ts` 中的某个 interface）
- [ ] **无共享状态修改** — 不同时修改同一个状态管理文件（如 Redux store、Zustand store、Context Provider）
- [ ] **无数据流依赖** — 子任务 A 的输出不是子任务 B 的输入

> **探索 Agent 的"共享依赖"输出**正是为这个 checklist 提供数据。Devin 在收到探索结果后，根据"共享依赖"节的信息来判断子任务是否真正独立。

#### 正确 vs 错误的独立性判断示例

**正确示例 — 可分解**：

需求：给系统添加"导出 CSV"功能 + 修复"删除确认弹窗"样式 bug

- 子任务 A：在 `services/export.ts` 新增 CSV 导出逻辑，在 `routes/export.ts` 新增 API 路由
- 子任务 B：修改 `components/ConfirmDialog.tsx` 的样式
- 独立性检查：无文件重叠、无 import 关系、无共享类型、无共享状态、无数据流依赖 → 全部通过，可分解

**错误示例 — 不可分解**：

需求：修改 OAuth 登录流程，添加账号选择步骤

- 子任务 A：修改 `auth/oauth.ts` 添加账号选择逻辑
- 子任务 B：修改 `components/OAuthDialog.tsx` 添加账号选择 UI
- 子任务 C：添加 `tests/oauth.test.ts`
- 独立性检查：
  - `OAuthDialog.tsx` import 了 `auth/oauth.ts` 的类型和函数 → **import 关系存在** → 不通过
  - 两者共享 `OAuthState` 类型定义 → **共享类型** → 不通过
  - UI 组件调用 auth 模块的函数，auth 模块的返回值驱动 UI 状态 → **数据流依赖** → 不通过
  - 测试依赖前两者的实现 → **数据流依赖** → 不通过
- 结论：不可分解，应在主上下文中顺序执行

> v1 中将此 OAuth 例子标记为"auth 和 UI 可并行"是错误的。OAuth 登录流程中 UI 和 auth 逻辑紧密耦合，修改一方几乎必然影响另一方的接口。

#### 分解流程

```
1. Devin 根据探索 Agent 返回的结构化报告，判断独立性（使用上方 Checklist）
2. 将通过独立性检查的子任务分组
3. 对每个子任务组，用 Agent 工具派实现 Agent 执行
   prompt 必须包含（见下方模板）：
   a. 子任务的具体修改说明
   b. 项目规范（来自 projects.json）
   c. 从探索结果中提取的关键代码片段（接口签名、类型定义）
   d. 当前分支名和工作目录
4. 独立的子任务 Agent 可以在同一条消息中派发（多个 Agent tool use）
   - 底层 Agent 工具标记为 isConcurrencySafe，同一消息中的多个 Agent 调用会并发执行
   - 但这是实现细节，分解决策不应依赖这一点
5. 有依赖关系的子任务 Agent 必须顺序执行
6. 所有实现 Agent 完成后，进入集成检查（见"集成检查"节）
```

#### 实现 Agent Prompt 模板

```markdown
你是 Devin 的实现 agent。完成以下子任务：

## 任务
{子任务描述，明确说明要实现什么功能/修复什么 bug}

## 目标文件
{要修改的文件路径列表}

## 项目规范
- commit 格式：{conventions.commitFormat}
- 代码风格：读已有代码保持一致

## 上下文（由 Devin 从探索结果中提取）
{只包含与本子任务直接相关的代码片段：}
- 相关接口签名：
  ```typescript
  {从探索 Agent 报告中提取的接口定义}
  ```
- 相关类型定义：
  ```typescript
  {从探索 Agent 报告中提取的类型}
  ```
- 修改切入点：{从探索 Agent 报告中提取的建议}

## 约束
- 只修改上方"目标文件"中列出的文件
- 不要新增或修改任何类型定义文件（共享类型由 Devin 统一处理）
- 改完后确认无语法错误（可运行 tsc --noEmit 或对应的类型检查命令）
- 不要提交 commit（由 Devin 统一提交）
- 如果发现需要修改目标文件之外的文件，停止并报告原因
```

### 与 SPEC-01（验证 Agent）的集成

分解后多个实现 Agent 完成的场景下，验证 Agent 的介入时机和方式：

#### 时序

```
实现 Agent A 完成 ─┐
实现 Agent B 完成 ─┤
实现 Agent C 完成 ─┘
        │
        ↓
  集成检查（Devin 主体执行）
  - git diff --stat 确认所有预期文件都被修改
  - 运行 build 确认编译通过
  - 运行 test 确认测试通过
        │
        ↓ build + test 全部通过
        │
  统一派发验证 Agent（SPEC-01）
  - 传入所有子任务的合并修改
  - 传入原始完整需求描述
  - 一次性验证全部修改的集成正确性
        │
        ↓ VERDICT
        │
  ┌─────┴─────┐
  PASS        FAIL
  │           │
  提交 PR     修复 → 重新验证（最多 2 轮）
```

#### 规则

1. **统一验证，不分别验证** — 不对每个实现 Agent 的产出单独派验证 Agent。原因：
   - 分别验证无法发现集成问题（子任务 A 和 B 各自正确但组合后冲突）
   - 减少验证 Agent 的调用次数，节省 token
2. **验证时机**：所有实现 Agent 完成 + 集成 build/test 通过后
3. **验证 Agent 的 prompt** 必须包含：
   - 原始需求描述（完整的用户需求原文）
   - `git diff` 的完整输出（包含所有子任务的修改）
   - 修改方案概述（Phase 2 确认的方案摘要，包括分解策略说明）
   - 子任务清单（列出每个实现 Agent 负责了什么，方便验证 Agent 理解修改结构）
4. **FAIL 后的修复**：Devin 主体负责修复，不再分解。修复后重新走"集成检查 → 验证 Agent"流程

#### 集成检查失败（build/test 不通过）

```
集成检查失败
    │
    ↓
Devin 分析失败原因（读 build/test 错误输出）
    │
    ├── 类型不匹配 → Devin 在主上下文中修复共享类型
    ├── import 冲突 → Devin 在主上下文中调整 import
    ├── 逻辑冲突   → Devin 在主上下文中合并修复
    │
    ↓ 修复后
重新运行 build/test → 通过后 → 派验证 Agent
```

### Agent 失败回退策略

实现 Agent 可能失败（超时、API 错误、Agent 自身判断无法完成任务）。定义明确的回退路径：

#### 失败类型与处理

| 失败类型 | 检测方式 | 处理策略 |
|---------|---------|---------|
| Agent 返回错误 | Agent 工具返回 is_error | 重试 1 次（相同 prompt）；仍失败 → 回退到 Devin 主体执行该子任务 |
| Agent 报告无法完成 | Agent 返回文本中包含"无法完成"/"blocked"类似表述 | Devin 主体读取 Agent 返回的原因，在主上下文中直接执行该子任务 |
| Agent 修改了不该修改的文件 | `git diff --name-only` 包含不在"目标文件"列表中的文件 | `git checkout -- {非目标文件}` 撤回非预期修改，检查目标文件修改是否仍有效 |
| Agent 超时 | Agent 工具执行超时 | 不重试；Devin 主体在主上下文中直接执行该子任务 |
| 部分 Agent 成功、部分失败 | 逐个检查 Agent 返回 | 保留成功 Agent 的修改，仅对失败的子任务执行回退 |

#### 回退原则

1. **保留成功的子任务** — 不因一个子任务失败而回滚所有修改
2. **回退 = Devin 主体直接执行** — 不再尝试分解，直接在主上下文中完成失败的子任务
3. **回退次数上限** — 同一个任务最多回退 2 个子任务。超过 2 个子任务失败说明分解策略本身有问题，Devin 应放弃所有分解产出，回到主上下文中重新执行整个任务
4. **回退后仍走完整流程** — 回退执行完成后，照常进入集成检查 → 验证 Agent

## 边界条件与风险

| 风险 | 应对 |
|------|------|
| 子任务之间实际有隐含依赖（Devin 判断失误） | 独立性 Checklist 降低误判概率；集成检查阶段的 build/test 兜底；发现冲突后 Devin 在主上下文中合并修复 |
| 探索 Agent 返回结果不够精确或遗漏共享依赖 | Devin 可追加具体问题再派一次探索 Agent；探索 Agent 的结构化模板强制要求输出"共享依赖"节 |
| 实现 Agent prompt 上下文不够（缺少关键信息） | 数据流规范要求 Devin 从探索结果中提取接口签名和类型定义；实现 Agent 发现信息不足时应停止并报告 |
| 实现 Agent 修改了不该修改的文件 | prompt 中明确约束"只修改目标文件"；集成检查时 `git diff --name-only` 比对预期文件列表 |
| 探索 Agent 读文件后 Devin 仍然自己去读（上下文悖论） | SKILL.md 中明确禁止此反模式；数据流图作为视觉提醒 |
| Agent 工具不可用或 API 限流 | 回退到 Devin 主体直接在主上下文中执行全部工作 |

## 验收标准

1. 修改 3+ 文件的任务中，探索阶段使用了探索 Agent 且返回了结构化报告
2. Devin 根据独立性 Checklist 做出分解/不分解决策，判断依据可追溯
3. 探索 Agent 返回后，Devin 主体没有再自己读探索 Agent 提到的文件
4. 实现 Agent 的 prompt 中包含从探索结果提取的接口签名/类型定义片段
5. 预期修改量 < 50 行或子任务数 < 3 的任务未被分解
6. 所有实现 Agent 完成后，走了集成检查 → 统一验证 Agent 的流程
7. Agent 失败时正确执行了回退策略，保留了成功子任务的修改
8. 汇报结果中包含分解策略说明（是否分解、分了几个子任务、验证结果）
