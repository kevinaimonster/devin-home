## 1. 问题陈述

当前 devin-dispatcher 的 DEVIN.md 维护指引过于模糊：

```
每次执行任务后检查是否有新知识需要写入 DEVIN.md：
- 新发现的模块 → 追加到模块地图
- 新的领域→文件映射 → 追加到映射表
- 踩过的坑 → 追加到开发规范
```

这导致几个问题：
1. **无分区结构** — 信息堆积成一坨，查找困难
2. **只追加不更新** — 过时信息（已删除的文件、已变更的命令）永远留着
3. **无去重** — 同一个知识点可能被写入多次
4. **格式不统一** — 每次写入的格式随机
5. **与 CLAUDE.md 边界不清** — 同一个知识可能同时出现在两个文件里

Claude Code 的 memory 系统提供了可借鉴的设计：
- **分类（user/feedback/project/reference）**：每种 memory 有明确的 when_to_save 和 what_not_to_save
- **去重**：写入前检查 existing memory files，"update an existing file rather than creating a duplicate"
- **主键**：每个 memory file 有唯一的 `name` frontmatter 字段，用于匹配和更新
- **清理过时**：通过 `type` 和 `description` 判断是否仍然相关
- **不保存可推导信息**：明确列出不应保存的内容（代码模式、架构、git 历史、CLAUDE.md 已有内容）

## 2. DEVIN.md 与 CLAUDE.md 的边界定义

### 职责划分

| 维度 | CLAUDE.md | DEVIN.md |
|------|-----------|----------|
| 维护者 | 人类开发者 | Devin 自动维护 |
| 内容性质 | 项目的通用开发规范 | Devin 在任务中积累的补充知识 |
| 权威性 | **权威来源**，Devin 必须遵守 | 辅助知识，可被 CLAUDE.md 覆盖 |
| 典型内容 | 代码风格、PR 规范、测试要求、架构决策 | 模块地图、领域映射、踩坑记录、环境细节 |

### 去重原则

**写入 DEVIN.md 之前，必须先读取项目的 CLAUDE.md**（如果存在），执行以下检查：

1. **CLAUDE.md 中已有的知识** → 不写入 DEVIN.md（例如：CLAUDE.md 已记录 `yarn build` 命令，DEVIN.md 不再重复）
2. **CLAUDE.md 中有更权威的版本** → 不写入 DEVIN.md（例如：CLAUDE.md 说"commit 用 conventional commits"，DEVIN.md 不再写一遍）
3. **CLAUDE.md 未覆盖的补充知识** → 写入 DEVIN.md（例如：CLAUDE.md 没提到 Java 子模块的构建顺序依赖，DEVIN.md 应记录）

> **一句话判断**：如果某条知识从 CLAUDE.md 中就能获得，不写入 DEVIN.md。DEVIN.md 只记录 CLAUDE.md 中找不到的、Devin 在实际开发中发现的补充知识。

## 3. DEVIN.md 模板结构

每个项目的 DEVIN.md 必须遵循以下**固定 5 分区**结构：

```markdown
# {项目名} — Devin 知识库

> 最后更新: {ISO 日期}

## 模块地图

项目顶层模块及其职责，每行一个。

- `frontend/` — React 前端，Next.js App Router
- `kernel/` — 核心业务逻辑，Spring Boot
- `common/` — 共享工具类和常量

## 领域 → 文件映射

业务领域到关键文件的映射，用于快速定位修改位置。

| 领域 | 关键文件 | 备注 |
|------|----------|------|
| OAuth 登录 | `auth/oauth.ts`, `components/OAuthDialog.tsx` | |
| 支付 | `biz/payment/`, `frontend/src/pages/pay/` | 需要 Stripe 测试 key |

## 构建 & 测试

验证命令，按执行顺序排列。

```bash
# 构建
yarn build                    # 或 mvn clean package

# 测试
yarn test --watchAll=false    # 或 mvn test -pl {module}

# 类型检查
yarn typecheck               # 或 tsc --noEmit

# Lint
yarn lint
```

## 踩坑记录

开发过程中发现的**非显而易见**的注意事项。每条必须有原因说明。

- **Java 子模块构建顺序**: 必须先 `mvn install -pl common` 再构建其他模块。原因：其他模块 pom.xml 依赖 common 的 SNAPSHOT 版本
- **env 文件**: `.env.local` 不在 git 中，首次 clone 需从 `.env.example` 复制。原因：包含 Stripe 测试 key

## 环境前置条件

运行本项目所需的环境依赖。

| 类别 | 要求 | 备注 |
|------|------|------|
| Node | >= 18.x | 使用 nvm 管理 |
| Java | JDK 17 | JAVA_HOME 需指向 17 |
| env var | `STRIPE_TEST_KEY` | 从 `.env.example` 获取 |
| 外部服务 | Redis 7.x | 本地开发需启动，`docker compose up redis` |
| 前置安装 | `brew install protobuf` | gRPC 代码生成依赖 |
```

### 各分区上限

| 分区 | 上限 | 超出时处理 |
|------|------|-----------|
| 模块地图 | 20 条 | 合并子模块到父模块描述 |
| 领域 → 文件映射 | **40 条** | 淘汰引用文件已不存在的条目 |
| 构建 & 测试 | 15 条命令 | 删除冗余的等价命令 |
| 踩坑记录 | 15 条 | 淘汰引用文件已不存在的条目 |
| 环境前置条件 | 15 条 | 合并同类项 |

## 4. 主键与去重

### 主键定义

v1 中使用"相似"来判断是否需要更新条目，但"相似"缺乏操作性定义。v2 为每个分区引入**主键**概念，主键相同即视为同一条目：

| 分区 | 主键 | 匹配规则 | 示例 |
|------|------|----------|------|
| 模块地图 | **目录路径** | 反引号中的路径去尾 `/` 后比较 | `frontend/` 和 `frontend/` 是同一条 |
| 领域 → 文件映射 | **领域名** | 领域列的文本完全匹配 | "OAuth 登录"和"OAuth 登录"是同一条 |
| 构建 & 测试 | **命令本体** | 去除注释后的命令文本匹配 | `yarn build` 和 `yarn build` 是同一条 |
| 踩坑记录 | **问题描述前 20 字** | `**` 加粗标题内的前 20 个字符 | "Java 子模块构建顺序"（14 字）匹配同一条 |
| 环境前置条件 | **类别 + 要求** | 表格前两列拼接后匹配 | "Node >= 18.x"和"Node >= 18.x"是同一条 |

### 去重操作

写入前必须先读取 DEVIN.md，按主键检查：

1. **主键不存在** → 新增条目
2. **主键存在且内容不同** → 用新内容**替换**旧条目（保留位置）
3. **主键存在且内容相同** → 跳过，不写入

## 5. 写入规则

### 5.1 什么时候写入

每次任务完成后（Phase 3.5 汇报结果之后），检查以下 5 个分区是否需要更新：

| 分区 | 触发条件 | 写入内容 |
|------|----------|----------|
| 模块地图 | 探索阶段发现了 DEVIN.md 中未记录的模块 | 模块路径（主键） + 一句话职责 |
| 领域 → 文件映射 | 本次任务涉及的"领域→文件"关系不在映射表中 | 领域名（主键） + 关键文件路径 |
| 构建 & 测试 | 使用了 DEVIN.md 中未记录的 build/test 命令 | 完整命令（主键） + 注释 |
| 踩坑记录 | 遇到了非显而易见的问题（如特殊构建顺序、隐含依赖） | 问题描述（主键） + 原因 |
| 环境前置条件 | 发现了运行项目所需但未记录的环境依赖 | 类别+要求（主键） + 备注 |

### 5.2 写入前置检查

```
写入流程：
1. 读取项目的 CLAUDE.md（如存在）
2. 读取项目的 DEVIN.md（如存在）
3. 对于每条待写入知识：
   a. 检查 CLAUDE.md 中是否已有 → 有则跳过
   b. 检查 DEVIN.md 中是否有同主键条目 → 有则更新，无则新增
4. 执行局部清理（见 6.1）
5. 执行结构校验（见 8）
6. 更新"最后更新"日期
```

## 6. 清理规则

### 6.1 局部清理（每次任务后）

**触发时机**：每次写入 DEVIN.md 时自动执行。

**范围**：只检查**本次任务涉及的分区条目**，不扫描整个 DEVIN.md。

具体操作：

| 条件 | 操作 |
|------|------|
| 本次涉及的模块地图条目 → 对应目录不存在 | 删除该行 |
| 本次涉及的文件映射条目 → 关键文件不存在 | 删除该行 |
| 本次涉及的踩坑记录 → 提到的问题已被修复 | 删除该条 |
| 本次涉及的环境前置条件 → 依赖已不需要 | 删除该条 |
| 构建命令在本次任务中执行失败且已被替代 | 更新为新命令 |

> **设计理由**：v1 要求每次写入时全量清理所有分区，但实际上 Claude 在 context 有限时会跳过大部分检查。局部清理只检查当前任务已经接触过的条目，工作量可控，执行率更高。

### 6.2 全量清理（显式触发）

**触发条件**（满足任一即触发）：

1. 用户明确要求："整理 DEVIN.md"、"清理 DEVIN.md"、"更新 DEVIN.md"
2. 局部清理时发现 > 30% 的已检查条目过时（推断整体可能也过时）

**全量清理流程**：

```
1. 读取完整 DEVIN.md
2. 模块地图：逐条检查目录是否存在 → 不存在则删除
3. 领域映射：逐条检查关键文件是否存在 → 文件全部不存在则删除该行
4. 构建命令：不主动验证（太耗时），保留现有条目
5. 踩坑记录：检查引用的文件/路径是否存在 → 全部不存在则标注"(待验证)"
6. 环境前置条件：保留（环境依赖不随代码删除而失效）
7. 读取 CLAUDE.md → 删除 DEVIN.md 中与 CLAUDE.md 重复的条目
8. 执行结构校验（见第 8 节）
9. 更新"最后更新"日期
```

## 7. 淘汰策略

### 基于文件存在性的淘汰（替代 v1 的"最不常用"）

v1 的淘汰策略要求"删除最旧/最不常用的条目"，但 DEVIN.md 中不存在访问频率数据，该策略无法执行。

v2 改为**基于文件存在性**的淘汰，可操作性强：

```
淘汰候选判定：
1. 模块地图条目 → 检查目录路径是否存在
   - 不存在 → 淘汰候选 → 直接删除
2. 领域映射条目 → 检查"关键文件"列中的文件是否存在
   - 全部不存在 → 淘汰候选 → 直接删除
   - 部分不存在 → 更新"关键文件"列，移除不存在的文件
3. 踩坑记录 → 检查条目中引用的文件路径是否存在
   - 引用的文件全部不存在 → 淘汰候选 → 标注"(待验证)"
   - 无文件引用的条目（纯经验型）→ 保留
4. 环境前置条件 → 不基于文件存在性淘汰（环境依赖独立于代码文件）
```

### 超出上限时的淘汰顺序

当某分区条目数超出上限时，按以下顺序淘汰：

1. **已标注"(待验证)"的条目** → 优先删除
2. **引用文件已部分不存在的条目** → 次优先删除
3. **内容最短/信息量最少的条目** → 最后删除

## 8. 结构校验

### 写入后校验

每次写入 DEVIN.md 后，检查以下结构完整性：

```
校验项（全部必须通过）：
1. 标题行：存在 `# {项目名} — Devin 知识库`
2. 更新日期：存在 `> 最后更新: {日期}`
3. 五个分区标题全部存在：
   - `## 模块地图`
   - `## 领域 → 文件映射`
   - `## 构建 & 测试`
   - `## 踩坑记录`
   - `## 环境前置条件`
4. 分区顺序正确（按上述顺序排列）
5. 各分区条目数不超过上限
```

### 校验失败处理

| 失败类型 | 处理 |
|----------|------|
| 缺少分区标题 | 在正确位置插入空分区（含标题和说明文字，无条目） |
| 分区顺序错误 | 重新排列分区 |
| 条目数超限 | 执行淘汰策略 |
| 标题行或日期缺失 | 补充 |

## 9. 迁移方案

### 检测条件

当读取项目的 DEVIN.md 后，发现文件**不包含 5 个标准分区标题中的任意 3 个**时，判定为非结构化 DEVIN.md，触发迁移。

### 迁移流程

```
1. 读取现有 DEVIN.md 全部内容
2. 读取项目 CLAUDE.md（如存在），用于去重
3. 逐段分析现有内容，按语义分类到 5 个分区：
   a. 提到目录结构、模块职责的 → 模块地图
   b. 提到"XX 功能在 YY 文件"的 → 领域 → 文件映射
   c. 提到 build/test/lint/typecheck 命令的 → 构建 & 测试
   d. 提到"注意"、"坑"、"必须先"、"不要"等警告性内容的 → 踩坑记录
   e. 提到 Node/Java/Python 版本、env var、docker、外部服务的 → 环境前置条件
   f. 无法归类的 → 放入"踩坑记录"末尾，标注 "(待归类)"
4. 与 CLAUDE.md 去重：已有的知识不写入
5. 按模板结构生成新的 DEVIN.md
6. 执行结构校验
7. 更新"最后更新"日期
```

### 迁移安全

- 迁移前**不删除**原文件，而是**覆写**（git 可追溯）
- 无法归类的内容标注"(待归类)"而非丢弃
- 迁移完成后在终端输出摘要：`已迁移 DEVIN.md：{N} 条模块地图 + {M} 条映射 + {K} 条踩坑 + {J} 条环境前置条件 + {L} 条待归类`

## 10. 新项目首次创建 DEVIN.md

当目标项目没有 DEVIN.md 时，在 Phase 1 探索完成后创建：

```
1. 读取项目 CLAUDE.md（如存在），记录其中已有的知识
2. 使用模板结构创建 DEVIN.md：
   a. 模块地图：从探索结果填充（排除 CLAUDE.md 已有的）
   b. 领域 → 文件映射：从本次任务涉及的领域填充
   c. 构建 & 测试：从 package.json scripts / pom.xml / Makefile 提取（排除 CLAUDE.md 已有的）
   d. 踩坑记录：留空
   e. 环境前置条件：从 package.json engines / .nvmrc / Dockerfile / docker-compose.yml 提取
3. 执行结构校验
```

## 12. 边界条件与风险

| 风险 | 应对 |
|------|------|
| DEVIN.md 太大（信息积累） | 各分区设上限（见第 3 节），超出时按淘汰策略清理 |
| 项目结构变化大（重构后大量过时） | 全量清理机制（见 6.2），用户也可手动触发 |
| 用户手动编辑了 DEVIN.md | 尊重用户的修改。主键匹配失败的条目视为用户新增，保留不动 |
| 多个 Devin 任务并发修改 DEVIN.md | 使用追加模式而非全量覆写；git merge 通常能处理追加冲突 |
| CLAUDE.md 不存在 | 跳过去重检查，正常写入 DEVIN.md |
| 迁移后丢失信息 | 无法归类的内容标注"(待归类)"而非丢弃 |
| 结构校验后分区顺序被打乱 | 校验包含顺序检查，自动修复 |
| 局部清理覆盖率不够 | 全量清理作为补充，用户可随时手动触发 |

## 13. 验收标准

1. 新创建的 DEVIN.md 遵循模板结构（**5 个分区**齐全）
2. 写入前读取了 CLAUDE.md 并执行了去重检查
3. 写入前按**主键**执行了去重（而非模糊的"相似"判断）
4. 每次任务后执行了**局部清理**（仅检查本次涉及的条目）
5. 全量清理仅在用户要求或检测到高比例过时时触发
6. 淘汰策略基于**文件存在性**检查（而非无法量化的"最不常用"）
7. 非结构化 DEVIN.md 被正确迁移到 5 分区结构
8. 写入后通过结构校验（5 个分区标题完整、顺序正确、条目不超限）
9. 各分区不超过上限（映射表上限 **40 条**）
10. 构建 & 测试分区命令**按执行顺序排列**（而非模糊的"按优先级"）
